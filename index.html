How Real Phone Lock Systems Work
Understanding the real-world mechanisms before building — from PIN hashing to biometric templates and secure enclave storage.
🔢
PIN Authentication
  • User enters 4–6 digit PIN on device
  • PIN is never stored in plain text
  • Hashed using PBKDF2 / bcrypt with device-specific salt
  • Hash is stored in Trusted Execution Environment (TEE)
  • On entry, hash of input compared to stored hash
  • Lockout after 5–10 failed attempts

👁
Biometric Authentication
  • Fingerprint: Capacitive sensor captures ridge pattern
  • Pattern converted to a mathematical template (not raw image)
  • Template stored in Secure Enclave (Apple) / TrustZone (Android)
  • Face ID: IR camera + dot projector maps 30,000 points
  • Neural engine computes 3D depth map and matches
  • Biometric never leaves the chip — comparison done on-device
🔐
Secure Storage
  • Secure Enclave — isolated coprocessor, separate memory
  • Keys generated inside enclave, never exposed to OS
  • PBKDF2/bcrypt/argon2 used to stretch PINs
  • Salt: unique random bytes per device/user
  • Iterations: 100,000+ rounds to slow brute force
  • Android Keystore / iOS Keychain for key management
python — hashing example
import bcrypt

# Store PIN
def hash_pin(pin: str) -> bytes:
    salt = bcrypt.gensalt(rounds=12)
    return bcrypt.hashpw(pin.encode(), salt)

# Verify PIN
def verify_pin(pin: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(pin.encode(), hashed)

Lockout Mechanism
  • Attempt counter increments on each failure
  • After 5 attempts → 30-second lockout
  • After 10 attempts → 1-hour lockout (progressive)
  • After N attempts → full wipe (optional)
  • Counter stored in persistent hardware memory
  • Counter resets ONLY on successful auth

🔢 How PIN Lock Works
1

User taps digits

OS captures keypress events. Raw digit is never logged.

2

PIN → Hash + Salt

bcrypt(PIN + salt, rounds=12). Takes ~300ms intentionally.

3

Hash comparison in TEE

Stored hash vs computed hash compared inside Secure Enclave.

4

Grant access or increment counter

Match → unlock. Fail → attempt counter +1. 5 fails = lockout.

⬡ How Pattern Lock Works
1

Touch events recorded

Screen tracks X,Y coordinates as finger moves across the 3×3 grid.

2

Node sequence extracted

Which dots were hit, in order → e.g. [1,2,3,5,8] = sequence string.

3

Sequence is hashed

SHA-1 / bcrypt of the node sequence string. Pattern = "12358" → hash.

4

Hash compared, access granted

Same as PIN — compare computed hash to stored hash. Never store the raw sequence.

Security note

Pattern has only 389,112 possible combinations (vs 10,000 for 4-digit PIN but with smudge-attack vulnerability). Android deprecated it in favor of PIN/biometric. Correct pattern: L-shape → nodes 1→4→7→8→9

👆 How Fingerprint Works
1

Capacitive sensor fires

500+ microscopic capacitors measure ridge vs valley at each point.

2

Minutiae extracted

Ridge endings, bifurcations mapped into a coordinate template. Never a raw image.

3

Template matching in Secure Enclave

Current template vs enrolled template using Euclidean distance with 0.1% FAR threshold.

4

Cryptographic key released

Match unlocks a private key which decrypts device storage. No match = key stays locked.

Key security facts

False Accept Rate: 1 in 50,000 (vs Face ID: 1 in 1,000,000). Template is stored encrypted in Secure Enclave — not in flash storage, not accessible by any app or OS process. Press & hold the sensor to try it!

auth.py — PIN Hashing with hashlib
auth.py
import hashlib, os, json
from config import STORAGE_FILE

def setup_pin(pin: str) -> None:
    """Hash PIN with salt and save to file."""
    salt = os.urandom(32).hex()
    hashed = hashlib.pbkdf2_hmac(
        'sha256', pin.encode(), bytes.fromhex(salt), 100_000
    ).hex()
    data = {"pin_hash": hashed, "salt": salt, "attempts": 0, "locked_until": None}
    json.dump(data, open(STORAGE_FILE, "w"))
    print("✔ PIN set successfully")

def verify_pin(pin: str, stored_data: dict) -> bool:
    """Re-hash input and compare with stored hash."""
    hashed = hashlib.pbkdf2_hmac(
        'sha256',
        pin.encode(),
        bytes.fromhex(stored_data['salt']),
        100_000
    ).hex()
    return hashed == stored_data['pin_hash']

lockout.py — Attempt Counter + Timeout
lockout.py
import time, json
from config import STORAGE_FILE, MAX_ATTEMPTS, LOCKOUT_SECS

def is_locked(data: dict) -> bool:
    if data.get("locked_until"):
        remaining = data["locked_until"] - time.time()
        if remaining > 0:
            print(f"🔒 Locked. Try again in {remaining:.0f}s")
            return True
        data["locked_until"] = None   # lockout expired
    return False

def record_failure(data: dict) -> None:
    data["attempts"] += 1
    remaining = MAX_ATTEMPTS - data["attempts"]
    if data["attempts"] >= MAX_ATTEMPTS:
        data["locked_until"] = time.time() + LOCKOUT_SECS
        print(f"🚨 Too many attempts! Locked for {LOCKOUT_SECS}s")
    else:
        print(f"❌ Wrong PIN. {remaining} attempts left.")
    json.dump(data, open(STORAGE_FILE, "w"))

def record_success(data: dict) -> None:
    data["attempts"] = 0
    data["locked_until"] = None
    json.dump(data, open(STORAGE_FILE, "w"))

main.py — Entry Point
main.py
import json, os
from auth    import setup_pin, verify_pin
from lockout import is_locked, record_failure, record_success
from config  import STORAGE_FILE, PIN_LENGTH

def main():
    # First run: set up PIN
    if not os.path.exists(STORAGE_FILE):
        pin = input("Set your PIN: ")
        setup_pin(pin)

    data = json.load(open(STORAGE_FILE))

    if is_locked(data):
        return

    pin = input("Enter PIN: ")
    if verify_pin(pin, data):
        print("✅ Unlocked! Welcome.")
        record_success(data)
    else:
        record_failure(data)

if __name__ == "__main__":
    main()

auth.py — bcrypt (Industry Standard)
app/auth.py
import bcrypt
from .models import db, Device, LoginHistory
from datetime import datetime

def hash_pin(pin: str) -> str:
    """bcrypt auto-generates and embeds salt in hash."""
    return bcrypt.hashpw(
        pin.encode('utf-8'),
        bcrypt.gensalt(rounds=12)
    ).decode('utf-8')

def verify_and_log(
    device_id: str, pin: str, ip: str
) -> dict:
    device = Device.query.filter_by(device_id=device_id).first_or_404()

    # Check lockout
    if device.locked_until and device.locked_until > datetime.utcnow():
        secs = (device.locked_until - datetime.utcnow()).seconds
        return {"success": False, "locked": True, "retry_in": secs}

    ok = bcrypt.checkpw(pin.encode(), device.pin_hash.encode())

    # Log the attempt
    db.session.add(LoginHistory(
        device_id=device_id, success=ok, ip_address=ip
    ))

    if ok:
        device.attempts = 0
        device.locked_until = None
    else:
        device.attempts += 1
        device.locked_until = _compute_lockout(device.attempts)

    db.session.commit()
    return {"success": ok, "attempts": device.attempts}

def _compute_lockout(attempts: int):
    from datetime import timedelta
    # Progressive: 30s → 5m → 30m → 1h → 24h
    schedule = [30, 300, 1800, 3600, 86400]
    if attempts < 5: return None
    idx = min(attempts - 5, len(schedule) - 1)
    return datetime.utcnow() + timedelta(seconds=schedule[idx])

jwt_handler.py — Access + Refresh Tokens
app/core/jwt_handler.py
from datetime import datetime, timedelta
from jose import jwt, JWTError
import os

SECRET_KEY     = os.getenv("JWT_SECRET")
ALGORITHM      = "HS256"
ACCESS_EXPIRE  = timedelta(minutes=15)   # Short-lived
REFRESH_EXPIRE = timedelta(days=7)       # Long-lived

def create_access_token(device_id: str) -> str:
    payload = {
        "sub"  : device_id,
        "type" : "access",
        "exp"  : datetime.utcnow() + ACCESS_EXPIRE
    }
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

def create_refresh_token(device_id: str) -> str:
    payload = {
        "sub"  : device_id,
        "type" : "refresh",
        "exp"  : datetime.utcnow() + REFRESH_EXPIRE
    }
    return jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)

def decode_token(token: str) -> dict:
    try:
        return jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid token")

otp_handler.py — TOTP / 2FA (Google Authenticator)
app/core/otp_handler.py
import pyotp, qrcode, io, base64

def generate_secret() -> str:
    """Generate unique TOTP secret per device."""
    return pyotp.random_base32()

def get_qr_code(secret: str, device_id: str) -> str:
    """Returns base64 QR code image for scanning in Google Auth."""
    uri = pyotp.totp.TOTP(secret).provisioning_uri(
        name=device_id, issuer_name="PhoneLock"
    )
    img = qrcode.make(uri)
    buf = io.BytesIO()
    img.save(buf, format="PNG")
    return base64.b64encode(buf.getvalue()).decode()

def verify_otp(secret: str, otp: str) -> bool:
    """Verify 6-digit TOTP code. Window=1 allows ±30s clock drift."""
    totp = pyotp.TOTP(secret)
    return totp.verify(otp, valid_window=1)

biometric.py — Face Detection Simulation (OpenCV)
app/core/biometric.py
import cv2, numpy as np, base64
from pathlib import Path

# Load Haar Cascade (ships with OpenCV)
FACE_CASCADE = cv2.CascadeClassifier(
    cv2.data.haarcascades + "haarcascade_frontalface_default.xml"
)

def detect_face(image_b64: str) -> dict:
    """Detect face in base64-encoded image."""
    img_data = base64.b64decode(image_b64)
    nparr   = np.frombuffer(img_data, np.uint8)
    img     = cv2.imdecode(nparr, cv2.IMREAD_COLOR)
    gray    = cv2.cvtColor(img, cv2.COLOR_BGR2GRAY)

    faces = FACE_CASCADE.detectMultiScale(
        gray, scaleFactor=1.1, minNeighbors=5, minSize=(100, 100)
    )
    return {
        "face_detected": len(faces) > 0,
        "face_count"   : len(faces),
        "confidence"   : 0.92 if len(faces) > 0 else 0.0
    }

def capture_intrusion_image(image_b64: str, device_id: str) -> str:
    """Save image on failed attempt for intrusion detection."""
    from datetime import datetime
    ts  = datetime.utcnow().strftime("%Y%m%d_%H%M%S")
    path = Path(f"intrusions/{device_id}_{ts}.jpg")
    path.parent.mkdir(exist_ok=True)
    img_data = base64.b64decode(image_b64)
    path.write_bytes(img_data)
    return str(path)

Security Considerations
What We Do Right
  • bcrypt with cost factor 12 — brute force resistant
  • JWT with short expiry (15 min access token)
  • Progressive lockout — slows attackers exponentially
  • Rate limiting at Nginx layer
  • TOTP 2FA adds second factor
  • Intrusion image capture on failure
  • Parameterized queries via SQLAlchemy (no SQL injection)
  • HTTPS only in production
Common Vulnerabilities to Know
  • Timing attacks on string comparison — use hmac.compare_digest
  • JWT secret leakage — rotate regularly, use env vars
  • PIN reuse across devices — add device-specific salt
  • Replay attacks on OTP — check used OTPs in Redis
  • Side-channel in bcrypt — use constant-time comparison
  • CSRF on web endpoints — use SameSite cookies
  • Insecure intrusion image storage — encrypt at rest

Common Interview Questions
Q1Why do we hash passwords/PINs instead of encrypting them?›

Encryption is reversible — if someone gets the key, they can decrypt all passwords. Hashing is a one-way function. We don't need to know the original PIN, we only need to verify it matches. When a user enters their PIN, we hash the input and compare hashes. bcrypt is ideal because it's deliberately slow (adaptive cost factor), has built-in salting to prevent rainbow table attacks, and the cost can be increased as hardware improves without changing existing hashes.

Key point: NEVER use MD5 or SHA-1 for passwords — they're too fast, making brute force trivial.

Q2What is the difference between Access Token and Refresh Token?›

Access Token: Short-lived (15 minutes). Sent with every API request. If stolen, damage is limited to 15 minutes. Stored in memory (not localStorage).

Refresh Token: Long-lived (7 days). Only sent to /auth/refresh endpoint. Stored in HttpOnly cookie. Used to get a new access token silently when it expires.

Why two tokens? Pure UX + security balance. You don't want users to log in every 15 minutes (bad UX), but you don't want a stolen token to last 7 days (bad security). The refresh token narrows the attack surface to a single endpoint.

Q3How does your lockout system prevent brute force attacks?›

Three-layer defense:

1. Progressive lockout — after 5 fails: 30s, then 5min, 30min, 1h, 24h. An attacker trying 1000 PINs would need weeks due to exponential delays.

2. Nginx rate limiting — blocks more than 10 requests/min per IP at the network layer, before reaching application code.

3. bcrypt cost factor — even if the hash leaks, cracking a single PIN takes seconds per attempt vs. microseconds with MD5. 100 rounds of bcrypt ≈ 300ms per hash.

Q4How does TOTP (Time-based OTP) work?›

TOTP is defined in RFC 6238. Both the server and the user's phone share a secret key. Every 30 seconds, a new 6-digit code is computed as: TOTP = HOTP(secret, floor(time/30)) where HOTP is HMAC-SHA1.

Since both sides know the secret and the current time, they independently compute the same code. No network exchange needed. The valid_window=1 in pyotp allows ±30 seconds for clock drift.

Important: each OTP must be marked as used in Redis to prevent replay attacks within the 30-second window.

Q5How does your biometric simulation work and how does real biometrics differ?›

Our simulation uses OpenCV's Haar Cascade classifier (machine learning-based pattern detection) to detect faces in a base64-encoded image. It returns a boolean + confidence score, which we treat as "authenticated".

Real biometrics (Face ID): Uses a structured light projector to create a 3D depth map of 30,000 points. A neural network converts this to a mathematical template. Authentication compares templates using Euclidean distance or cosine similarity — never comparing raw images. The template never leaves the Secure Enclave chip.

Key difference: We simulate the flow (capture → detect → authenticate), not the cryptographic security of real biometric systems.

Q6Why use Redis for lockout state instead of PostgreSQL?›

Lockout state needs to be checked on every authentication request — potentially thousands per second at scale. Redis is an in-memory store with O(1) GET/SET and sub-millisecond latency vs. PostgreSQL's 1–5ms network round-trip per query.

Redis also has built-in TTL (expiry), perfect for: SET lockout:device123 1 EX 1800 — automatically expires after 30 minutes, no cleanup job needed.

PostgreSQL is still the source of truth — Redis is the performance layer.

Q7What design patterns did you use in this project?›
  • Repository pattern — data access logic separated from business logic (DB queries in models/db layer)
  • Strategy pattern — authentication method is pluggable: PIN, Biometric, OTP each implement a common interface
  • Dependency Injection — FastAPI's Depends() for DB sessions and current user extraction
  • Factory pattern — Flask/FastAPI app factory for testing with different configs
  • Observer pattern — login events trigger side effects (logging, intrusion capture) decoupled from auth logic
Q8How would you scale this system to 10 million devices?›
  • Horizontal scaling of FastAPI with Kubernetes + HPA (auto-scale on CPU/RPS)
  • PostgreSQL read replicas for analytics queries — write to primary, read from replicas
  • Redis Cluster for distributed lockout state across multiple Redis nodes
  • Kafka/SQS for async event processing — login events → analytics pipeline
  • CDN + S3 for intrusion image storage (pre-signed URLs)
  • Database sharding by device_id if PostgreSQL hits limits
  • HSM (Hardware Security Module) for production key management instead of env vars
  • Migrate bcrypt to Argon2id — better memory-hard, recommended by OWASP
